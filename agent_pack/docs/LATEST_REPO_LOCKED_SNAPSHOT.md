## 3. التعديلات اليدوية التي تم فهمها وتسجيلها

- النظام الحالي مبني على قاعدة بيانات Fresh، وليس ترحيل بيانات قديمة.
- واردات الكتب / استلام المخزون هي **مخزون فقط** وليست ماليات.
- الدفع ليس تقسيطاً؛ يوجد دفع جزئي أو دفع كامل أو فاتورة غير مدفوعة/آجلة.
- التوريد المالي يعني تحويل مبلغ محصل إلى الخزينة، وليس توريد كتب.
- المدفوعات تدعم رفع إيصال/مرفق مع العملية.
- التطبيق الحالي يتعامل مع الإيصالات كـ `approved` مباشرة في كود إنشاء الدفع والفاتورة، مع بقاء backend endpoints للمراجعة موجودة.
- تم إزالة Review Queue من واجهة المدفوعات حسب تقرير Step 148، رغم أن بعض سياسات الـ Agent Pack القديمة كانت تطلب Queue.
- الشحن الجزئي موجود على مستوى بنود الفاتورة وكمياتها.
- الاسترجاع موجود على مستوى بنود الفاتورة، ويزيد المخزون ويضيف قيد مالي بقيمة سالبة على receivable.
- الفواتير تدعم Actions من القائمة/التفاصيل: دفع، توريد، شحن، استرجاع، عرض/تصدير حسب الصلاحيات.
- التصدير الحالي CSV احترافي؛ Import ممنوع في الواجهة الحالية.

## 4. قواعد البيزنس الحالية المقفولة

### الفواتير والدفع

- حالات الدفع المعروضة يجب أن تبقى: غير مدفوع/آجل، مدفوع جزئياً، مدفوع بالكامل.
- لا يوجد تقسيط ولا Payment Plans.
- كل دفع له مبلغ وطريقة دفع وتاريخ ومرجع وملاحظات وحالة توريد.
- `supply_status = supplied` يعني تم توريد المبلغ للخزينة.
- `supply_status = not_supplied` يعني تم التحصيل لكن لم يورد للخزينة.

### الماليات

- الرصيد المعلق = المتبقي على المنافذ بعد المدفوع والمسترجعات.
- الرصيد المحصل = إجمالي المدفوعات المقبولة.
- المحصل المورد = مدفوعات supply_status = supplied.
- المحصل غير المورد = مدفوعات supply_status = not_supplied.
- رصيد المسترجعات يدخل في كشف حساب المنفذ والليميت.

### المخزون

- واردات الكتب تزيد المخزون فقط.
- بيع فاتورة ينقص المخزون.
- الاسترجاع يزيد المخزون.
- التسويات تعدل المخزون بسجل مستقل.
- لا يوجد أي تأثير مالي مباشر لواردات الكتب.

### الشحن

- الشحن يتم باختيار بنود من الفاتورة وكميات محددة.
- لا يمكن شحن أكثر من المتبقي غير المشحون وغير المرتجع.
- الشحن الجزئي لا يغير الماليات.

### الاسترجاع

- الاسترجاع لا يحذف الفاتورة ولا الدفع.
- الاسترجاع يسجل `return` + `return_items`.
- الاسترجاع يزيد مخزون المنتجات المتتبعة.
- الاسترجاع يقلل Exposure/Receivable عبر قيد مالي.

## 6. ملاحظات تقنية مهمة لا يجب نسيانها

- `server/config/index.js` يحسب `rootDir` بـ `../../..` من `server/config`، وهذا يخرج لمجلد أعلى من المشروع. عند العمل لاحقاً يجب مراجعته؛ غالباً الهدف الصحيح `../..`.
- رفع الإيصالات داخل `server/modules/invoices/invoicesService.js` و `server/modules/payments/paymentsService.js` يستخدم path ثابت:
  - `D:/Projects/BookStore Manager/storage`
  وهذا غير مناسب للـ DirectAdmin ويجب تحويله إلى config/uploadsDir إذا تم فتح Step جديدة.
- Fresh schema لا يزال يحتوي على `import_jobs` و `import_job_rows` في migration 001 رغم منع Import في المنتج الحالي.
- Migration 004/008 فيها بقايا category باسم `installment_due` في notifications schema، رغم أن التقسيط ملغي. خدمة الإشعارات الحالية لا تسمح بها في `allowedCategories`، لكن قاعدة البيانات لا تزال تحتويها.
- بعض Docs القديمة داخل التقارير التاريخية تتكلم عن مراحل/قرارات تم تجاوزها. المصدر الحالي للحقيقة هو هذا الملف + `status.json` بعد قفله.
