## 1. الخطوة الأولى: تحميل الكود وتشغيل التثبيت التلقائي

بما أنك متصل بالفعل بالخادم وتوجد في المجلد الرئيسي للـ VPS، يمكنك تنزيل الكود وتشغيل التنصيب بالكامل **بأمر واحد فقط**:

```bash
git clone https://github.com/ziadelsayed2004/hamza.printing.press.git hamza-press && cd hamza-press && chmod +x deploy.sh && sudo ./deploy.sh
```

> [!NOTE]
> إذا كنت قد قمت برفع الملفات يدوياً مسبقاً (عبر `rsync` أو `scp`) وتوجد بالفعل داخل مجلد المشروع، فقط قم بتشغيل السكربت مباشرة:
> ```bash
> chmod +x deploy.sh && sudo ./deploy.sh
> ```

## 2. الخطوة الثانية: ماذا يفعل سكربت التثبيت التلقائي؟

عند تشغيل السكربت [deploy.sh](file:///deploy.sh)، سيقوم بالعمليات التالية تلقائياً دون أي تدخل منك:

1. **تثبيت حزم بيئة التشغيل:** تنصيب وتحديث `Node.js v20`, `Nginx`, `SQLite3`, `Git`, `PM2`, و `UFW Firewall`.
2. **طلب نطاق العمل (Domain Name):** سيسألك السكربت عن اسم النطاق الموجه للسيرفر:
   * **إذا لم يتوفر لديك نطاق حالياً:** اضغط على زر **[ENTER]** وسيقوم السكربت بالاستعلام عن الـ IP الخارجي للسيرفر وتعيينه تلقائياً.
3. **توليد البيئة الإنتاجية (.env):** نسخ ملف `.env.example` وإنشاء مفتاح تشفير قوي وعشوائي لـ `SESSION_SECRET` وتهيئة التطبيق على وضع الإنتاج (`production`).
4. **تنزيل حزم البرمجة وبناء الواجهات:** تثبيت حزم `npm` للواجهة الخلفية والأمامية معاً وتصدير ملفات React لتعمل بأقصى سرعة بكفاءة (Vite Build).
5. **تهيئة قاعدة البيانات:** إنشاء قاعدة بيانات SQLite وتفعيل الجداول وإدخال بيانات الصلاحيات والمدير الافتراضي (Super Admin).
6. **تأمين الاتصال (Firewall):** إعداد جدار الحماية UFW لفتح منافذ الويب والـ SSH فقط.
7. **إعداد الـ Reverse Proxy (Nginx):** ربط منفذ التطبيق الداخلي (5000) بالمنفذ الخارجي العام (80) لتصفح الموقع مباشرة.
8. **إعداد شهادة الأمان (SSL / HTTPS) تلقائياً:** يسألك السكربت إذا كنت ترغب في تثبيت شهادة Let's Encrypt وتفعيل HTTPS بشكل تلقائي وآمن وتوجيه جميع الطلبات تلقائياً إليه.
9. **جدولة النسخ الاحتياطي التلقائي (Cron):** جدولة عملية أخذ نسخة احتياطية آمنة لقاعدة البيانات يومياً الساعة 2:00 صباحاً.

## 3. الخطوة الثالثة: إدارة قاعدة البيانات وصلاحيات الوصول (SQLite)

قاعدة بيانات النظام مبنية على **SQLite** وهي عبارة عن ملف محلي مستقل داخل المجلد المعزول `storage/`:
`/var/www/hamza-press/storage/database.sqlite`

يقوم السكربت بضبط صلاحيات الملفات والمجلدات لضمان أقصى حماية برمجية مع تمكين خادم Node.js من القراءة والكتابة بحرية داخل مجلد التخزين:
```bash
# تغيير ملكية مجلد الويب بالكامل للمسؤول
chown -R root:root /var/www/hamza-press

# إعطاء صلاحيات الكتابة للتطبيق داخل مجلد التخزين
chmod -R 777 /var/www/hamza-press/storage
```

## 4. الخطوة الرابعة: التحكم في خادم التطبيق ومراقبته (PM2)

تُدار عملية الخلفية الخاصة بالنظام بواسطة مدير العمليات **PM2**، مما يضمن إعادة تشغيل التطبيق ذاتياً في حال تعطل السيرفر أو حدوث خطأ مفاجئ.

استخدم هذه الأوامر البسيطة للتحكم في خادم التطبيق:
```bash
# لعرض حالة العمليات المفعّلة واستهلاك الرامات والمعالج:
pm2 status

# لمشاهدة سجلات وتفاصيل العمليات الحالية مباشرة:
pm2 logs

# لإعادة تشغيل النظام بالكامل لتطبيق تحديثات برمجية جديدة:
pm2 restart bookstore-manager

# لإيقاف تشغيل الخادم مؤقتاً:
pm2 stop bookstore-manager
```

## 5. الخطوة الخامسة: نظام النسخ الاحتياطي التلقائي (Database Backups)

لحماية بيانات مطبعتك من الفقدان، تم إعداد نظام نسخ احتياطي تلقائي ثنائي الطبقات:

### أ. الطبقة الأولى: النسخ الاحتياطي التلقائي من النظام (Cron Job - مستحسن وآمن جداً)
تم جدولة مهمة (Cron Job) لتشغيل سكربت آمن [backup-db-safe.sh](file:///scripts/backup-db-safe.sh) **يومياً الساعة 2:00 صباحاً**.
* يضمن استخدام أمر `.backup` المدمج في SQLite3 نسخ قاعدة البيانات بشكل متوافق وتجنب أي تلف حتى أثناء كتابة البيانات من قبل المستخدمين.
* يقوم السكربت تلقائياً بحذف النسخ الاحتياطية التي مضى عليها أكثر من **30 يوماً** للحفاظ على مساحة القرص.
* تُحفظ النسخ في المسار المخصص: `/var/www/hamza-press/storage/backups/`.

### ب. الطبقة الثانية: جدولة التطبيق الداخلية (Node.js Scheduler)
يعمل كود إضافي داخل التطبيق [backupScheduler.js](file:///server/modules/admin/backupScheduler.js) لمحاولة أخذ نسخة احتياطية يومية إضافية كخط دفاع ثانٍ في حال تعطل خدمة Cron بالنظام.

### ج. أخذ نسخة احتياطية يدوية فورية:
إذا أردت إجراء نسخة احتياطية فورية يدوياً في أي وقت، قم بتشغيل هذا الأمر داخل مجلد المشروع:
```bash
npm run backup
```
أو تشغيل السكربت مباشرة:
```bash
bash scripts/backup-db-safe.sh
```

## 6. الخطوة السادسة: كيفية استعادة قاعدة البيانات من نسخة احتياطية (Restore)

في حال حدوث خطأ أو رغبتك في العودة بالبيانات إلى تاريخ سابق، يمكنك استعادة أي نسخة احتياطية بأمان عبر اتباع الخطوات التالية بالتوالي:

1. **إيقاف تشغيل التطبيق مؤقتاً (هام جداً لمنع تعارض الكتابة):**
   ```bash
   pm2 stop bookstore-manager
   ```
2. **عرض النسخ الاحتياطية المتوفرة واختيار الملف المناسب:**
   ```bash
   ls -la /var/www/hamza-press/storage/backups/
   ```
3. **نسخ ملف النسخة الاحتياطية ليحل محل قاعدة البيانات النشطة:**
   *(استبدل اسم الملف بالاسم الفعلي للنسخة التي تريد استعادتها)*
   ```bash
   cp /var/www/hamza-press/storage/backups/backup_2026-07-09_02-00-00.sqlite /var/www/hamza-press/storage/database.sqlite
   ```
4. **تحديث صلاحيات ملف قاعدة البيانات للتأكد من قدرة التطبيق على القراءة والكتابة فيه:**
   ```bash
   chmod 777 /var/www/hamza-press/storage/database.sqlite
   ```
5. **إعادة تشغيل خادم التطبيق مرة أخرى:**
   ```bash
   pm2 start bookstore-manager
   ```

## 7. الخطوة السابعة: تحديث النظام مستقبلاً وسحب التعديلات (Seamless Updates)

بما أن قاعدة البيانات والملفات المرفوعة والنسخ الاحتياطية تعيش جميعها داخل مجلد `storage/` وهو مجلد مُستثنى تماماً من تتبع الجيت لقفل البيانات، يمكنك تحديث الكود وسحب الميزات الجديدة من GitHub في أي وقت دون الخوف من فقدان أو الكتابة فوق بياناتك.

لتحديث النظام، قم بتشغيل الأوامر التالية بالتوالي داخل مجلد المشروع `/var/www/hamza-press/`:

```bash
# 1. سحب التعديلات البرمجية الجديدة من GitHub
git pull origin main

# 2. تثبيت وتحديث الموديولات البرمجية الجديدة للـ backend والـ client
npm install
npm install --prefix client

# 3. إعادة بناء وتصدير ملفات واجهة المستخدم (Vite Build)
npm run build

# 4. تطبيق أي تحديثات جديدة على هيكل الجداول البرمجية بقاعدة البيانات (Migrations)
npm run db:migrate

# 5. إعادة تشغيل التطبيق في PM2 لتفعيل التعديلات
pm2 restart bookstore-manager
```

## 8. الخطوة الثامنة: استعراض السجلات وتتبع الأخطاء (Logs & Debugging)

إذا واجهت أي مشاكل في عمل التطبيق أو التصفح، يمكنك استعراض سجلات أخطاء خادم الويب Nginx أو التطبيق مباشرة عبر الأوامر التالية:

* **عرض سجل أخطاء خادم Nginx:**
  ```bash
  tail -n 50 -f /var/log/nginx/error.log
  ```
* **عرض سجل وصول الطلبات لخادم Nginx:**
  ```bash
  tail -n 50 -f /var/log/nginx/access.log
  ```
* **عرض سجل تفاصيل خادم Node.js النشط:**
  ```bash
  pm2 logs bookstore-manager --lines 100
  ```

### ⚠️ استكشاف الأخطاء وإصلاحها: فشل تصدير ملفات الـ PDF (خطأ Puppeteer/Chromium)
إذا ظهر لك خطأ 500 عند تصدير الفواتير إلى ملف PDF، ووجدت في سجلات الأخطاء رسالة مثل:
`error while loading shared libraries: libnss3.so: cannot open shared object file`
هذا يعني أن نظام التشغيل Ubuntu على السيرفر يفتقد لبعض المكتبات المشتركة اللازمة لتشغيل متصفح Chromium المدمج في Puppeteer.

**الحل:**
قم بالاتصال بالسيرفر عبر SSH وتشغيل الأمر التالي لتثبيت كافة المكتبات اللازمة:
```bash
sudo apt-get update && sudo apt-get install -y libnss3 libnspr4 libatk1.0-0 libatk-bridge2.0-0 libcups2 libdrm2 libxkbcommon0 libxcomposite1 libxdamage1 libxrandr2 libgbm1 libasound2
```
بعد انتهاء التثبيت، أعد تشغيل التطبيق لتطبيق التغييرات:
```bash
pm2 restart bookstore-manager
```
